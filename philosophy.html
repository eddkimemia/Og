MANIFESTO

MY 5 RULES FOR CODE

I build software that is meant to be read, maintained, and evolved. Here is the philosophy that guides every line I write.

  1. Readability over cleverness: The next engineer (including future me) should understand this in 30 seconds. One-liners are for vanity; clarity is for production.
  2. Type safety by default: TypeScript strict mode is non-negotiable. I use runtime validation (Zod) at all API boundaries to ensure the "outside world" doesn't break my app.
  3. Test behavior, not implementation: I prioritize integration tests that mimic user interactions over brittle unit tests that break every time I refactor a function name.
  4. Fail loudly in dev, gracefully in prod: Use meaningful error messages and structured logging. Never swallow an error silently.
  5. Optimize for change: Code will be read 10x more than written. I structure modules so that features can be added or removed without a house-of-cards collapse.
