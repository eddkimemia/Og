Debugging a Race Condition in a Node.js WebSocket Server – A War Story

It started with a few reports of "missing messages" in our real-time chat application. Under normal load, everything was fine. But during peak hours, messages were seemingly vanishing into the void.

[ERROR] WebSocket: Failed to broadcast to client_8291. Socket state: CLOSING
[DEBUG] Event Loop Lag: 142ms

The Symptoms

The logs showed that we were trying to send messages to clients that were in the process of disconnecting. However, the application state still thought they were active. It was a classic race condition between the `disconnect` event and the `broadcast` loop.

The False Leads

I initially thought it was a memory leak or a backpressure issue with the underlying TCP stream. We wasted hours tuning the `highWaterMark` and optimizing the JSON serialization logic, but the drops persisted.

The Actual Bug

The issue was that our `activeClients` map was being updated asynchronously. A broadcast could start, yield the event loop, and by the time it resumed, the client reference it held was stale but not yet removed from the map.

The Fix: Async Queue

We implemented a robust async queue to handle outgoing messages. Instead of iterating the map directly during a broadcast, we pushed messages to a worker that validated the socket state immediately before transmission.

// The Fix: Using a Queue to serialize broadcasts
const broadcastQueue = async.queue(async (task) => {
  const { socket, message } = task;
  if (socket.readyState === WebSocket.OPEN) {
    await socket.send(message);
  }
}, 100); // Concurrency limit

The Victory

After deploying the fix, our message delivery reliability went from 94% to 99.8%. The event loop lag dropped significantly because we were no longer blocking it with massive, un-throttled iteration loops.
