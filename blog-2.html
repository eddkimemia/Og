INTRODUCTION

We were seeing a 6% message drop on our WebSocket server during peak hours. No errors in the logs, just missing data. It was a classic race condition, and it was driving us crazy.

THE DEEP DIVE

The issue was multiple clients updating the same task simultaneously. Our database was overwriting state based on whichever request finished last, regardless of the sequence of the user actions. We needed a reliable way to queue and process these updates sequentially.

In a distributed system, you can't assume order unless you enforce it.

CODE IMPLEMENTATION

async function processMessage(msg) {
  const lock = await redis.set(msg.id, 'locked', 'NX', 'EX', 5);
  if (!lock) return reschedule(msg);
  try {
    await updateDB(msg);
  } finally {
    await redis.del(msg.id);
  }
}

LESSONS LEARNED

Always use idempotency keys for critical state changes. It's the only way to sleep soundly at night when you're handling thousands of concurrent users.
