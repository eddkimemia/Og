INTRODUCTION

As our dataset grew, a critical dashboard query began taking over 5 seconds to execute. This was unacceptable for our users, and I was tasked with bringing it under 500ms.

THE DEEP DIVE

I started by analyzing the execution plan with EXPLAIN ANALYZE. It revealed multiple sequential scans on large tables. By adding targeted composite indexes and refactoring the query to use CTEs and window functions, I was able to optimize the data retrieval path.

Don't guess why a query is slow. Let the database tell you exactly where the bottleneck is.

CODE IMPLEMENTATION

EXPLAIN ANALYZE
SELECT u.name, count(o.id) OVER (PARTITION BY u.id)
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE o.created_at > NOW() - INTERVAL '30 days';

LESSONS LEARNED

Query optimization is an iterative process. Small changes in how you structure your joins can have massive impacts on performance as data volume scales.
